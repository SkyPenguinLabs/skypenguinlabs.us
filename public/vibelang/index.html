OVERVIEW

The language is a complete product surface, not just syntax.

Source Model

V.I.B.E. programs are processed through a conventional but inspectable pipeline: source text, lexer, parser, canonical AST, compiler, bytecode module, and VM/runtime output.

  • Language behavior is tied to audit corpus evidence.
  • Syntax profiles and execution profiles are part of the surrounding system.
  • Docs, examples, skills, tests, and runtime behavior are expected to stay aligned.

Runtime Model

The runtime centers on explicit values, environments, modules, plugins, native integration, taint/safety policy, scheduler behavior, traces, reports, and output contracts.

  • Runtime-facing changes require tests or audit proof.
  • Diagnostic output is treated as part of the product contract.
  • Unsafe capabilities are gated by execution profile flags.

Tooling Model

The CLI is the user-facing boundary. Help metadata, command manifests, config profiles, benchmarks, logs, indexes, and workbench behavior all feed the same language surface.

  • vibe help explain exposes command, flag, module, builtin, keyword, topic, config, and environment metadata.
  • Workbench and regular source files keep newline-delimited parser behavior.
  • One-liner semicolon normalization is scoped to explicit one-liner mode.

RUNTIME

Designed around evidence and explainability.

The V.I.B.E. repo treats language runs, behavioral changes, syntax changes, runtime capability changes, safety changes, and toolchain changes as one product surface. That means a runtime feature is not finished just because it executes: it also needs evidence in tests, audit corpus cases, examples, documentation, or skill updates.

The broad mastery path is source model first, execution model second, runtime integration third, then evidence through audit corpus, fuzz seeds, examples, benchmarks, docs, and release governance.

std.userlandreserved userland branch for higher-level runtime-facing helpers

Registration

register("std") exposes only the root topic modules. register("std/<topic>") binds one aggregate module with shared helpers plus nested child modules.

Removed Flat Paths

Flat native paths such as std/io, std/algebra, std/png, and std/error are removed with no compatibility aliases.

Gated Branches

std/web3 is exposed only behind compile.enable_blockchain_types. std/net/raw requires runtime.enable_raw_network.

NATIVE NETWORKING V1

HTTP-first, with lower-level lanes available.

The current native networking surface is HTTP-first. std/net/http includes client helpers plus router, route, server startup, directory serving, file serving, listen/accept/respond helpers, request and response parsing, raw request and response helpers, stats, events, and response builders for JSON, text, files, directories, and redirects.

TCP supports configurable listeners, accept deadlines, read/write deadlines, socket options, half-close helpers, byte send/receive, and address metadata. Proxy support provides managed reverse proxy handles with route metadata, stats, and event rings. Testing helpers are localhost-oriented.

INSTALLER

Linux-only in this workspace.

The installer detects Debian/Ubuntu, Arch-family, or generic Linux from /etc/os-release. Interactive installs may prompt before invoking package managers. Non-interactive runs and --yes do not auto-install packages; they print the exact package command and fail fast.

If a source refresh fails during --artifact source --update-source, the installer can fall back to the local checkout when go.mod, cmd/vibe, and .vibe_lang_store are present.

STATUS

Current state and boundaries.

Available Now

  • Explicit command suite under the vibe binary.
  • Terminal-aware help with --color auto|always|never and --no-color.
  • One-liner flags under vibe run: -e, -n, -p, -a, -F, -0, -i, -M, --begin, --end, and --dump-expanded-source.
  • Native networking V1 with HTTP, TCP, proxy, testing, and gated raw networking.

Intentional Removals

  • No direct vibe <file> execution form.
  • No implicit no-argument REPL startup.
  • No top-level operational flags, top-level --help, --ui-repl, or separate vibe-bench binary.
  • vibe help serve and older internal docserver routes are retired.

R&D Caveat

  • The language is still a lab-based research and development experiment.
  • It is used in field operations and production-facing applications, but should not be treated as broadly stable.
  • The checkout currently notes a docs/examples gap; runtime-facing changes should be reflected in README, help/catalog metadata, skills, and audit corpus until those trees are restored.
  • Reserved branches stay out of the runtime catalog until implemented.
